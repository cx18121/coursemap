## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, inline chip padding (`px-1`, `py-0.5`) |
| sm | 8px | Compact element spacing, toggle label gap (`gap-2`) |
| md | 16px | Default element spacing, card padding (`p-4`) |
| lg | 24px | Section padding (`p-5`, `p-6`) |
| xl | 32px | Layout gaps between dashboard sections (`space-y-8`) |
| 2xl | 48px | Major section breaks |
| 3xl | 64px | Page-level top padding (`pt-12` = 48px — nearest value) |

Exceptions: Touch targets for the toggle switch must be minimum 44px tall (accessibility). Toggle container uses `min-h-[44px]` with vertically centered content.

## Color

Sourced from `src/app/globals.css` `@theme` block. No new color tokens introduced in this phase.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#0c0a09` (`--background`) | Page background |
| Secondary (30%) | `#18181b` (`--color-surface`) + `#1e1e22` (`--color-surface-raised`) | Card surfaces, toggle container background (`bg-white/10 backdrop-blur-lg`) |
| Accent (10%) | `#f59e0b` (`--color-accent`) | Focus ring on toggle switch, active toggle thumb |
| Destructive | `red-500/10` bg + `red-400` text | Error states only (load error pattern already established) |

Accent reserved for: focus rings (`*:focus-visible` outline), active toggle switch thumb/track when type grouping is enabled. NOT used for toggle labels, section headers, or informational chips.

The type grouping toggle uses `accent-indigo-500` for the HTML checkbox (matching the existing course checkbox pattern), NOT `--color-accent`. This preserves visual consistency with existing checkboxes in `CourseAccordion`.

## Component Inventory

### New: TypeGroupingToggle

Location: `src/components/TypeGroupingToggle.tsx`

A global opt-in switch displayed in `SyncDashboard.tsx` above the Canvas Courses section header.

**Visual anatomy:**
```
┌─────────────────────────────────────────────────────────┐
│  [toggle switch]  Group events by type                  │
│                   Each course gets sub-calendars:        │
│                   Assignments · Quizzes · Discussions    │
└─────────────────────────────────────────────────────────┘
```

- Container: `bg-white/10 backdrop-blur-lg rounded-2xl border border-[--color-border] p-4`
- Layout: `flex items-center gap-3 min-h-[44px]`
- Toggle: HTML `<input type="checkbox">` with `className="w-4 h-4 rounded accent-indigo-500 flex-shrink-0 cursor-pointer"` (matches existing course checkbox pattern)
- Label: `text-sm font-medium text-[--color-text-primary]`
- Sub-label (descriptive line): `text-xs text-[--color-text-secondary]` — shown below label
- Type chips row: `flex gap-1.5 flex-wrap mt-1` — displays `Assignments · Quizzes · Discussions · Events` as passive text chips using `text-xs text-[--color-text-tertiary] bg-white/10 rounded-full px-2 py-0.5`

**States:**
- Unchecked (default): Toggle off, sub-label visible, chips dimmed (`text-[--color-text-tertiary]`)
- Checked (active): Toggle on, sub-label visible, chips use `text-[--color-text-secondary]`
- Saving (optimistic): Toggle flips immediately (optimistic update), no spinner. If PATCH fails, toggle reverts with no toast (silent failure acceptable for MVP — matches existing course toggle pattern)

**Placement in SyncDashboard:** Between the page header block and the Canvas Courses `<section>`. Only rendered when `hasCanvasUrl && courses.length > 0`.

### Extended: CourseAccordion

No visual changes to `CourseAccordion.tsx` for MVP. Type grouping is a back-end routing concern. The accordion continues to show events flat (not grouped by type within the accordion).

Future enhancement: per-course type grouping toggle as a chevron-adjacent icon. Out of scope for this phase.

## Interaction Contracts

### Type Grouping Toggle — On/Off

| Trigger | Response | Feedback |
|---------|----------|---------|
| User clicks toggle (off → on) | Optimistic state flip; PATCH `/api/user-settings` `{ typeGroupingEnabled: true }` | Toggle switches immediately. No loading state. |
| User clicks toggle (on → off) | Optimistic state flip; PATCH `/api/user-settings` `{ typeGroupingEnabled: false }` | Toggle switches immediately. No loading state. |
| PATCH succeeds | State stays as set | No additional feedback |
| PATCH fails | Revert toggle to previous state | No toast — silent revert (matches existing course/event toggle failure pattern) |
| Type grouping enabled + user syncs | Sync runs normally; progress indicators unchanged | SyncButton and SyncProgress show per-course progress (not per-type). No change to sync feedback UI. |

### SyncButton — No Changes

The SyncButton, SyncSummary, and SyncProgress components require no changes. Type grouping is transparent to the sync feedback layer. The summary still shows aggregate `inserted / updated / skipped / failed` counts.

## States and Edge Cases

| State | UI Treatment |
|-------|-------------|
| Type grouping off (default) | Toggle unchecked. Sync behaves identically to Phase 2. No visual change to CourseAccordion. |
| Type grouping on, first sync pending | Toggle checked. No new UI element. User syncs normally. |
| Type grouping on, 0 events of a type | No sub-calendar created server-side. No empty-type chip shown. Chips in TypeGroupingToggle are static (not dynamically updated). |
| Type grouping on, classification fallback | Events with unrecognized titles route to "Events" sub-calendar. No UI indication of classification result. |
| Toggle loading state | Optimistic only — no spinner. Same pattern as `handleToggleCourse` in `SyncDashboard.tsx`. |

## Accessibility

- Toggle checkbox has `aria-label="Enable event type grouping"` and `id="type-grouping-toggle"` with a `<label htmlFor="type-grouping-toggle">` wrapper.
- Touch target: container `min-h-[44px]` ensures 44px minimum hit area.
- Focus ring: `*:focus-visible` from globals.css applies `outline: 2px solid var(--color-accent)` automatically.
- Type chips are decorative (`aria-hidden="true"`) — they are not interactive.
- First-enable note uses `role="note"` for screen reader discoverability.
