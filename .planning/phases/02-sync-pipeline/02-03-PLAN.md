**Refactor parse-ics/route.ts:**
Change from POST accepting `{ feedUrl }` in body to GET using session auth:
1. Call `getSession()` — return 401 if null
2. Query `users` table for `canvasIcsUrl` by session.userId
3. If `canvasIcsUrl` is null, return 400 with `{ error: "No Canvas feed URL configured" }`
4. Call `parseCanvasFeed(user.canvasIcsUrl)` to get GroupedEvents
5. Call `ensureCourseSelections(userId, Object.keys(groupedEvents))` to upsert any new courses
6. Call `assignCourseColors(userId, Object.keys(groupedEvents))` to ensure colors are assigned
7. For each event, look up cleaned title from titleCleanup: call `cleanTitlesBatch(allTitles)` and attach `cleanedTitle` to each event in the response
8. Return `{ courses: Array<{ courseName, colorId, enabled, events: Array<{ uid, summary, cleanedTitle, description, start, end, excluded }> }> }`
9. The `excluded` boolean comes from eventOverrides (true if override exists with enabled=false)
10. Remove the old POST handler entirely

**Create user-selections/route.ts:**
- GET: Return all courseSelections and eventOverrides for the session user
  - `getSession()` — 401 if null
  - Query courseSelections WHERE userId = session.userId
  - Query eventOverrides WHERE userId = session.userId
  - Return `{ courseSelections: [...], eventOverrides: [...] }`

- PUT: Accept batch updates to selections
  - Body: `{ courseSelections?: Array<{ courseName, enabled?, colorId? }>, eventOverrides?: Array<{ eventUid, enabled?, customTitle? }> }`
  - For each courseSelection update: upsert into courseSelections (update enabled and/or colorId)
  - For each eventOverride update: upsert into eventOverrides
  - Return `{ success: true }`
  - Validate that colorId is between '1' and '11' if provided

**Create school-calendars/route.ts:**
- GET: Call `listSchoolCalendars(session.userId)`, return the array
  - If no school account linked (getFreshAccessToken returns null), return `{ calendars: [], linked: false }`
  - Otherwise return `{ calendars: SchoolCalendar[], linked: true }`

- PUT: Toggle school calendar selection
  - Body: `{ calendarId: string, calendarName: string, enabled: boolean }`
  - Upsert into schoolCalendarSelections
  - Return `{ success: true }`

All routes: Use `getSession()` for auth, return 401 when no session. No raw access tokens in request/response bodies.

<task type="auto">
  <name>Task 2: Create unified sync endpoint with polling-based progress</name>
  <files>src/app/api/sync/route.ts, src/app/api/sync/status/route.ts</files>
  <action>
**sync/route.ts (POST):**
1. `getSession()` — 401 if null
2. Generate a unique jobId (crypto.randomUUID())
3. Store job status in a module-level Map (in-memory, acceptable for manual sync — no concurrent sync jobs per user): `syncJobs: Map<string, { status: 'running' | 'complete' | 'error', progress: SyncProgress[], canvasSummary?: SyncSummary, mirrorSummary?: MirrorSummary, error?: string }>`
4. Start the sync in the background (do NOT await — return jobId immediately):
   a. Parse Canvas feed: call `parseCanvasFeed(user.canvasIcsUrl)`
   b. Filter events: call `filterEventsForSync(userId, groupedEvents)`
   c. Get color map: call `assignCourseColors(userId, courseNames)`
   d. Sync Canvas: call `syncCanvasEvents(userId, filteredEvents, colorMap, (p) => { store progress })`
   e. Mirror school: call `mirrorSchoolCalendars(userId, (p) => { store progress })`
   f. Mark job complete with summaries
   g. On error: mark job as error with message
5. Return `{ jobId }` with 202 Accepted

IMPORTANT: Use `waitUntil` from `next/server` if available in Next.js 16, or use a detached promise pattern. The key is the sync runs server-side after the HTTP response is sent. If `waitUntil` is not available, use a simple fire-and-forget promise (the sync function catches its own errors).

**sync/status/route.ts (GET):**
1. `getSession()` — 401 if null
2. Read `jobId` from URL search params
3. Look up in the syncJobs Map
4. If not found, return 404
5. Return the current job state: `{ status, progress, canvasSummary, mirrorSummary, error }`
6. If status is 'complete' or 'error', clean up the Map entry after returning (or set a 5-minute TTL)

The dashboard UI (Plan 04) will poll GET /api/sync/status?jobId=X every 500ms until status is 'complete' or 'error'.

NOTE: The in-memory Map means progress is lost on server restart, which is fine for a manual sync button. If the sync takes <15 seconds and the user stays on the page, polling works reliably.
